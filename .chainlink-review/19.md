# Issue #19: Phase 3: AppView & Index Aggregation Service

## REVIEW

We will still want to create an AppView for a number of reasons -- we should go through and talk about the design goals here, and create a gh issue laying this out.

## Verdict: ARCHIVE

## Evidence
The original vision for a standalone AppView service that aggregates records from the ATProto firehose has been superseded by the pluggable provider architecture. Instead of building a separate aggregation service, the project implemented pluggable index backends (SQLite, Redis, PostgreSQL) that can each serve as the persistence layer, paired with the `Repository` dataclass for composable multi-source dataset management. The `_AtmosphereBackend` adapter in `repository.py` provides ATProto routing without needing a separate AppView. The sub-issues (#32-#35) for firehose ingestion, search API, and caching were never implemented and are no longer needed given this architecture.
